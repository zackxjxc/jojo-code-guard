# 兼容性改进计划

本文档记录 Claude 和 Codex 自动守护链路的后续改进建议。以下内容是待实现计划，不代表当前插件已经具备这些能力；
当前实际行为以 [生效与验收.md](生效与验收.md) 为准。

## 目标与边界

- 不以升级 Codex 或 Claude 为前提，优先兼容当前已验证的 Codex 0.142.3 和 Claude Code 2.1.195。
- AI 写文件前遵守主 Skill，写入后立即检查，回合结束前补一次兜底检查，提交时可由 `pre-commit` 做最终机械门禁。
- 检查只报告和阻断污染，不自动格式化、转码、统一换行或覆盖用户修改。
- 主 Skill 不接管 `git commit`，也不规定是否允许 `git push`；提交和推送权限由用户指令及上层规则决定。
- 插件升级和客户端升级是独立事项。仅修改本仓库不会改变已安装缓存中的插件版本。

## 当前待解决问题

### 1. `PostToolUse` 阻断语义不利于 AI 自动修复

当前 `post-write-check` 在发现阻断项时返回 `continue: false` 和 `stopReason`。Claude 会停止本轮处理，且
`stopReason` 主要面向用户；Codex 对相同字段的处理方式不同。这会造成“发现污染但 AI 没有继续修复”的风险。

推荐改为两端都能理解的阻断反馈：

```json
{
  "continue": true,
  "decision": "block",
  "reason": "写入后检查发现阻断项，请先修复污染再继续",
  "hookSpecificOutput": {
    "hookEventName": "PostToolUse",
    "additionalContext": "完整诊断"
  }
}
```

实现时不得把已完成的写入描述成已回滚；Hook 只阻止后续步骤并把诊断交给 AI。无诊断时保持静默，协议错误继续
fail closed。

### 2. `SessionStart` 上下文输出需要统一

当前 `session-start` 直接输出普通 stdout，Claude 和 Codex 都支持把它注入会话上下文；Codex 还支持
`hookSpecificOutput.additionalContext`。推荐优先输出结构化 `SessionStart` JSON，使事件类型和上下文边界更明确，
并在两个客户端分别做真实会话验收。
如果某一端不能可靠解析同一结构，应由同步脚本生成客户端专用适配器，不在同一 stdout 中混合普通文本和 JSON。

Skill 文件无法读取时也要使用客户端可见的结构化上下文报告失败，并要求暂停文件修改，不能静默降级。

### 3. 编辑工具匹配不能覆盖全部写入路径

当前 `PostToolUse` 主要匹配编辑类工具，不能可靠覆盖 Bash、PowerShell、外部脚本和客户端新增的写入工具。建议分两层处理：

1. 对客户端已知且稳定的写入工具补齐 matcher；对可能写文件的 shell 工具，允许执行一次轻量差异检查。
2. 增加 `Stop` Hook，在 AI 回合准备结束时运行一次 `check_diff.py`。仅发现 `BLOCKED` 时返回诊断并要求继续修复；
   输入的 `stop_hook_active` 已标记为重入时直接放行，避免无限循环。

`Stop` 只作为遗漏路径的兜底，不能替代写入后的即时检查，也不能替代可选的 Git `pre-commit`。

### 4. Claude 与 Codex 的 Hook 清单应按客户端适配

Codex 支持 Windows 专用命令字段，Claude 当前公开清单字段与其不完全一致。长期建议由同步脚本从共同配置生成两份清单：

- Claude 清单只使用 Claude 官方支持的字段和工具名。
- Codex 清单使用 Codex 支持的 Windows 命令、工具 matcher 和生命周期事件。
- 两份清单调用同一组核心检查脚本，避免复制守护逻辑。
- Windows 路径包含空格、反斜杠或非 ASCII 字符时，不依赖业务仓库 cwd 或未定义的环境变量。

在拆分完成前，文档和 doctor 不得把“客户端发现了清单”表述成“Hook 一定已经执行”。

### 5. doctor 需要同时验收两端

doctor 目前应继续保持默认只读，并补充以下检查：

- 源码插件版本与 Claude、Codex 各自安装缓存版本是否一致。
- Codex 是否启用插件和 Hook 功能；若无法确定 Hook 信任状态，明确标为需要人工确认。
- Claude 是否启用插件、加载的 Hook 清单是否来自预期版本。
- 两端同步产物是否与源码 Hook、Skill 和脚本一致。
- 生命周期 Hook 的发现、信任和实际执行分别报告，不合并成一个“已启用”结论。

doctor 只给出升级、重装或信任步骤；没有用户确认时，不修改全局配置、不安装插件、不升级客户端。

## 推荐实施顺序

1. 调整 `PostToolUse` 阻断输出并补齐两端协议测试。
2. 将 `SessionStart` 改为结构化上下文，验证启动、恢复、清空和压缩会话。
3. 增加带重入保护的 `Stop` 兜底检查，并验证 shell 或外部脚本写入路径。
4. 生成 Claude/Codex 专用 Hook 清单，覆盖 Windows 路径和环境变量差异。
5. 扩展 doctor，对源码、安装缓存、功能开关、信任状态和实际执行分别验收。
6. 同步 README、主 Skill、usage、CHANGELOG 和插件版本，再发布并重新安装插件。

## 验收标准

- 正常写入且无诊断时，`PostToolUse` 和 `Stop` 不向会话制造噪声。
- 制造编码、BOM、换行、替换字符、Git 空白或异常 diff 污染后，AI 能收到完整诊断并继续修复，而不是无反馈地终止。
- 通过编辑工具、shell 和外部脚本制造污染时，最迟在回合结束前被发现。
- `Stop` Hook 不会因自身阻断反馈反复触发而形成循环。
- `SessionStart` 在两端都能注入完整 Skill；资源缺失时能明确暂停并报告。
- Hook 从业务仓库 cwd 启动时，仍能在 Windows、macOS 和 Linux 找到插件资源。
- Windows 插件路径包含空格、反斜杠和中文时，命令及 JSON 均不乱码。
- Claude 和 Codex 的 manifest validator、同步产物测试、Hook 协议测试及完整测试套件全部通过。
- 安装了 `pre-commit` 的仓库仍能拦截暂存区污染；未安装时不影响主 Skill、写入后检查和回合结束检查。
- 所有检查均不自动转码、格式化或改变 Git 状态，diff 中没有整文件重排和无关修改。

## 当前环境说明

制定本计划时，本机 PATH 中的 Codex 为 0.142.3，Claude Code 为 2.1.195；用户已决定暂不升级两个客户端。
因此实现和验收必须先覆盖这两个版本。较新客户端的修复可以作为额外兼容性验证，不能替代当前版本验收。
